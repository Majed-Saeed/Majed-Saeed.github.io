3:T564,<p>Losing SSH to a running instance feels like losing the instance. It isn&#39;t. If the workload is healthy and only the access path is broken, you can repair it by treating the root disk as data.</p>
<h2>The idea</h2>
<p>An EBS root volume is just a disk. Detach it from the broken instance, attach it to a working one, fix the file, and put it back.</p>
<h2>The steps</h2>
<ol>
<li>Stop the affected instance (don&#39;t terminate it).</li>
<li>Detach its root EBS volume.</li>
<li>Attach that volume to a temporary helper instance <strong>in the same availability zone</strong>.</li>
<li>Mount it and correct the offending file:</li>
</ol>
<pre><code class="hljs language-bash">sudo <span class="hljs-built_in">mkdir</span> -p /mnt/rescue
sudo mount /dev/xvdf1 /mnt/rescue
<span class="hljs-comment"># fix sshd_config, authorized_keys, or permissions under /mnt/rescue</span>
sudo umount /mnt/rescue
</code></pre><ol start="5">
<li>Detach from the helper, reattach to the original instance as its root device, and start it.</li>
</ol>
<h2>What to watch</h2>
<p>The availability-zone constraint trips people up — a volume can only attach to an instance in its own AZ. Device names and the exact partition to mount are the other easy mistakes.</p>
<p>The real win is having written the procedure down <em>before</em> you need it. An outage is a bad time to be improvising.</p>
